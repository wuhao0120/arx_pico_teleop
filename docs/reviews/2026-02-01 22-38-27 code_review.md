## 概述

ARX LIFT 实现是一个从 UR5e 转换到 ARX R5 双臂 + LIFT 底盘的良好设计。架构正确地避免了重复的 CAN 连接——通过让 teleoperator 共享 robot 的 BimanualArm 引用。但是，我发现了 **3 个严重问题**、**5 个主要问题** 和若干小问题需要在生产使用前解决。

## 严重问题 (必须修复)

### 1. 🔴 夹爪位置未从硬件读取

**位置:** `arx_lift.py:246-247`

**当前代码:**
```python
obs_dict["left_gripper_position"] = self._left_gripper_position
obs_dict["right_gripper_position"] = self._right_gripper_position
```

**问题描述:**
观测值使用的是内部命令变量 (`self._left_gripper_position`)，而不是从机器人读取实际夹爪状态。这意味着：
- 如果夹爪命令执行失败，记录的数据将是错误的
- 训练的策略会从错误的 ground truth 学习
- 没有反馈回路验证夹爪是否真正移动

**UR 对比:**
UR 实现使用 `self._left_gripper.pos` 从硬件读取实际位置。

**修复方案:**
从 `get_joint_positions()` 读取，它返回 7 个关节，索引 6 是夹爪：
```python
# 在 get_observation() 中，left_pos 和 right_pos 已经读取了
# 直接使用 left_pos[6] 和 right_pos[6]
obs_dict["left_gripper_position"] = left_pos[6]  # 第7个关节是夹爪
obs_dict["right_gripper_position"] = right_pos[6]
```

**影响:** 数据质量、策略训练准确性

### 2. 🔴 底盘高度启动时未同步

**位置:** `teleop_arx.py:152`

**当前代码:**
```python
self._current_height = 0.0  # 始终从 0.0 开始
```

**问题描述:**
teleop 的内部高度状态始终从 0.0 开始，但实际底盘可能处于不同高度。第一个高度命令将是 `0.0 + delta`，可能导致底盘意外移动。

**风险:**
⚠️ **安全隐患** - 当 VR 控制开始时，底盘可能移动到非预期高度。

**修复方案:**
在 `_init_qpos()` 中同步高度：
```python
def _init_qpos(self):
    left_pos = self._robot.bimanual_arm.get_joint_positions("left")["left"]
    right_pos = self._robot.bimanual_arm.get_joint_positions("right")["right"]
    self.target_left_q = np.array(left_pos)
    self.target_right_q = np.array(right_pos)

    # 新增：同步底盘高度
    self._current_height = self._robot.lift_controller.get_height()
    self.target_chassis_height = self._current_height
```

**影响:** 安全性、底盘控制准确性

### 3. 🔴 Episode 之间底盘高度未重置

**位置:** `teleop_arx.py:419-425`

**当前代码:**
```python
def reset_tracking(self):
    """Reset VR tracking state for all arms."""
    for arm_name in self.manipulator_config.keys():
        self.init_ee_xyz[arm_name] = None
        self.init_ee_quat[arm_name] = None
        self.init_controller_xyz[arm_name] = None
        self.init_controller_quat[arm_name] = None
    # 没有底盘高度重置！
```

**问题描述:**
当 `reset_tracking()` 在 episode 之间被调用时，手臂跟踪被重置但底盘高度状态保持不变。这导致累积式高度控制从上一个 episode 的高度继续。

**修复方案:**
添加底盘高度重置：
```python
def reset_tracking(self):
    for arm_name in self.manipulator_config.keys():
        self.init_ee_xyz[arm_name] = None
        self.init_ee_quat[arm_name] = None
        self.init_controller_xyz[arm_name] = None
        self.init_controller_quat[arm_name] = None

    # 新增：重置底盘高度到当前机器人状态
    if self._robot is not None and self._robot.lift_controller is not None:
        self._current_height = self._robot.lift_controller.get_height()
        self.target_chassis_height = self._current_height
```

**影响:** Episode 间数据一致性、底盘控制

## 主要问题

### 4. 🟠 重置时直接修改内部状态

**位置:** `run_record_arx.py:169`

**当前代码:**
```python
robot._chassis_height = record_cfg.init_height
```

**问题描述:**
直接设置内部状态而不是通过 `send_action()` 发送命令。这绕过了正常的命令流程，并与后台 LIFT 控制线程产生竞争条件。

**建议:**
使用 `send_action()` 发送命令，或至少添加同步机制。

---

### 5. 🟠 SDK IK 求解器 API 使用需验证

**位置:** `teleop_arx.py:315`

**当前代码:**
```python
ik_result = bimanual.inverse_kinematics(target_xyzrpy)
```

**问题描述:**
代码导入 `bimanual` 模块并直接调用 `inverse_kinematics()`。需要验证这个 API 是否真的是纯计算函数，不需要 CAN 连接。

**风险:**
如果此函数需要 CAN 连接，会破坏共享 robot 引用的设计初衷。

**建议:**
验证 `bimanual.inverse_kinematics()` 函数是纯计算的，无硬件依赖。

### 6. 🟠 关节数组无验证

**位置:** `arx_lift.py:188-189`

**当前代码:**
```python
left_pos = [action[f"left_joint_{i+1}.pos"] for i in range(self._num_joints)]
```

**问题描述:**
没有验证：
- `self._num_joints` 是否正确 (7)

### 7. 🟠 IK 失败只在 DEBUG 级别记录

**位置:** `teleop_arx.py:329-330`

```python
except Exception as e:
```

## 接口对齐分析 (UR RTDE vs ARX SDK)

| 方面 | UR (RTDE) | ARX (SDK) | 状态 |
|------|-----------|-----------|------|
| 连接方式 | 以太网/RTDE | CAN 总线 | ✅ 不同但正确 |
| 关节读取 | `rtde_receive.getActualQ()` | `bimanual.get_joint_positions()` | ✅ 对齐 |
| 关节写入 | `rtde_control.servoJ()` | `bimanual.set_joint_positions()` | ✅ 对齐 |
| 夹爪控制 | 串口 (pyDHgripper) | SDK `set_catch_pos()` | ⚠️ 观测未从硬件读取 |
| 末端位姿 | RTDE + FK | `get_ee_pose()` | ✅ 对齐 |
| 逆运动学 | Placo (外部) | SDK 内置 | ✅ 方法不同但正确 |
| 遥测数据 | RTDE 读取 | CAN 读取 | ✅ 对齐 |

## 架构验证

### 当前架构 (正确)

```
VR 输入 (XrClient)
    ↓
ARXVRTeleop._process_arm_vr()
    ├─ 从 robot.bimanual_arm 读取当前状态
    ├─ 使用 bimanual.inverse_kinematics() 计算 IK
    ├─ 更新 self.target_left_q, self.target_right_q
    ├─ 不发送命令到机器人
    └─ 通过 get_action() 返回

    ↓
record_loop() (LeRobot 基类)
    ├─ 调用 teleop.get_action()
    ├─ 处理 action
    ├─ 调用 robot.send_action()
    └─ 记录到数据集

    ↓
ARXLift.send_action()
    ├─ 通过 bimanual_arm.set_joint_positions() 发送关节命令
    ├─ 通过 bimanual_arm.left_arm.set_catch_pos() 发送夹爪命令
    ├─ 更新底盘速度变量
    └─ 后台线程执行底盘命令
```

**验证结果:** ✅ 没有像最初设计那样的重复命令问题。

## 总结

ARX LIFT 实现代表了一个**设计良好的架构重构**。使用 robot 引用而非重复连接、使用 SDK IK 而非 Placo 的决定都是合理的。

但是，**三个严重问题**（夹爪状态观测、底盘高度同步、episode 重置）可能影响数据质量和安全性，应在生产使用前解决。

代码有良好的文档，包含线程、配置管理和日志的良好实践。测试基础设施比原始 UR 实现更好。修复推荐的问题后，这应该是一个健壮的 ARX LIFT 数据收集系统。
